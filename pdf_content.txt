## 1. Domain and URL consistency (mandatory) 
 
**Problem:** Repo currently assumes `https://www.meeruai.com` while 
production will be **`https://meeru.ai`** . 
 
**Agency shall:** 
 
| # | Deliverable | Acceptance criterion | 
|---|-------------|----------------------| 
| 1.1 | Update **Next.js metadata** (`metadataBase`, Open Graph URL, Twitter 
card base URLs as applicable) so all generated absolute URLs use 
**`https://meeru.ai`** . | View page source / social debugger: `og:url` and related 
tags point to **meeru.ai** . | 
| 1.2 | Update **`public/robots.txt`** so `Sitemap:` references 
**`https://meeru.ai/sitemap.xml`** . | robots file fetched from production/staging 
shows correct host. | 
| 1.3 | Update **`public/sitemap.xml`** so every `<loc>` uses 
**`https://meeru.ai`** and matches **real routes** deployed in production. | 
Sitemap validates; no leftover `www.meeruai.com` URLs. | 
| 1.4 | Search and replace remaining **`meeruai.com`** references in shipped 
content (e.g. legal pages, footer, body copy) so they match **business-approved** 
wording (apex `meeru.ai` vs legacy domain). Client may specify exact legal text. | 
Repo search: zero unintended old-domain references **or** an explicit list of 
exceptions approved by client. | 
 
**Note:** Client configures **301 redirects** at DNS/CDN (e.g. `www.meeru.ai` ŌåÆ 
`meeru.ai`, or old domain ŌåÆ new). Agency documents **which URL the app treats

## 2. SEO metadata per route (mandatory) 
 
**Problem:** Only the root layout may export global `metadata`; inner pages risk 
inheriting the **homepage** title/description. 
 
**Agency shall:** 
 
| # | Deliverable | Acceptance criterion | 
|---|-------------|----------------------| 
| 2.1 | Add route-level **`metadata` (or `generateMetadata`)** for at least: `/`, 
`/about`, `/privacy`, `/terms`, `/trust`. | Each URL has **unique** `<title>` and 
meta description suitable for search/social sharing. | 
| 2.2 | Ensure **Open Graph / Twitter** fields are correct for key pages if the client 
requires share cards per page (minimum: unique title/description; images may 
inherit global if agreed). | Validated with Meta/Twitter debug tools on staging. |

## 3. Security headers (optional ŌĆö policy-dependent) 
 
This is a **static marketing site** (no first-party API surface for pages). **HTTPS** 
and the **baseline headers** already set in `next.config.ts` are the practical 
minimum for production. 
 
**HSTS** and **Content-Security-Policy (CSP)** are **defense in depth** , not 
something you ŌĆ£mustŌĆØ have simply because the app is static. CSP in particular 
needs staging validation with Next.js, Formspree, fonts, and any analytics ŌĆö so 
treat it as a **follow-up hardening** task unless **your organizationŌĆÖs security or 
compliance policy** explicitly requires it. 
 
| # | Item | Guidance | 
|---|------|----------| 
| 3.1 | **HSTS** | **Recommended** at CDN/load balancer (or via Next headers) 
when production is **100% HTTPS** , so browsers always use TLS. | 
| 3.2 | **CSP** | **Optional** default; **mandatory only if client policy says so. ** If 
implemented, test on staging first; expect iteration for script/style/connect 
allowances. | 
 
Ownership: **agency or infrastructure** ŌĆö document who sets edge vs app 
headers.
